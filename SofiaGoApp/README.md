# SofiaNow

SofiaNow е Expo/React Native приложение за придвижване в София, което комбинира карта в реално време, разписания, trip planner, любими места, локални напомняния и режим за паркиране. Кодът е организиран като мобилно приложение с отделен web вариант и използва публични транспортни данни от София Трафик/ЦГМ, статични GTFS ресурси и локално съхранение на устройството.

## Кратко резюме за потребители

Със SofiaNow можеш бързо да провериш къде се намира градският транспорт в момента, кога идва на спирката ти и кой е най-удобният маршрут до желаната дестинация. Приложението обединява карта в реално време, близки спирки, пълни разписания, търсене на адреси и линии, както и запазване на любими места и маршрути.

Освен транспортните функции, приложението предлага локални напомняния за тръгване и пристигане, преглед на паркинги и паркинг зони в София, както и изпращане на parking SMS на Android. Идеята е на едно място да имаш както ежедневното придвижване с транспорт, така и основните инструменти за паркиране в града.

## Какво прави приложението

Приложението има два основни режима:

- `Transit` режим за градски транспорт: карта, спирки, линии, ETA, разписания, търсене, trip planner, любими места и напомняния.
- `Parking` режим за паркиране: синя/зелена зона, платени паркинги, мои автомобили и изпращане или планиране на parking SMS на Android.

## Подробна функционалност

### 1. Карта в реално време

Картата е централният екран на приложението.

- Показва текущата локация на потребителя и може да центрира изгледа върху нея.
- Показва спирки, линии и превозни средства в реално време.
- Зарежда превозни средства само за видимата част от картата, за да пази производителността.
- Поддържа избор на спирка, избор на превозно средство и визуализиране на конкретен маршрут върху картата.
- Поддържа dropped pin с дълго натискане върху картата, откъдето може да се стартира търсене на маршрут или да се запази място в любими.
- Позволява превключване между транспортен и паркинг режим, като home action bar-ът сменя действията си според режима.

Какво вижда потребителят на картата:

- активни автобуси, трамваи, тролеи и метро
- спирки с линии и подробности
- маршрутна геометрия за избрана линия или trip planner маршрут
- филтри по тип транспорт и по линия
- панел с действия за изчистване на избран маршрут, превозно средство или активна зона

Технически бележки:

- Native вариантът използва `MapLibre`, а web вариантът използва `Leaflet`.
- Данните за превозни средства и ETA идват от GTFS Realtime потоци на София Трафик.
- Част от overlays и floating controls са специфични за mobile UI.

### 2. Превозни средства в реално време

Приложението следи позициите на превозните средства и ги анимира плавно върху картата.

- Обновява позициите приблизително на всеки 1.5 секунди.
- Изчислява heading/посока на движение и визуализира иконите според типа транспорт.
- Позволява избор на конкретно превозно средство и показване на неговия маршрут.
- Съчетава филтрите по линия и тип транспорт с визуализацията на превозните средства.

### 3. Спирки и ETA информация

Спирките са една от най-пълно реализираните части на приложението.

- Зареждат се от статични ресурси, bundled в приложението.
- За избрана спирка могат да се покажат live ETA времена.
- За всяка спирка могат да се виждат линии, посоки и детайлно разписание.
- При избор на спирка от списък или от карта приложението може да фокусира картата върху нея.
- От спирка могат да се създават любими места и напомняния за пристигане.

Ограничения:

- ETA данните са live и зависят от публичния GTFS-RT feed.
- Част от детайлите за разписанията имат fallback към bundled статични JSON данни.

### 4. Екран „Около мен"

Екранът `Nearby` използва GPS позицията на устройството и групира близките спирки по пеша достъпност.

- Разделя спирките на 5, 10 и 15 минути пеша.
- Показва разстояние до спирката и основните линии.
- При разгъване на спирка зарежда live ETA данни за нея.
- Позволява добавяне или премахване на спирката от любими.
- Дава бързо действие за външна навигация или за отваряне на маршрут към избраната спирка.

Ограничения:

- Работи коректно само при разрешен достъп до локация.
- По подразбиране разглежда радиус до около 15 минути пеша.

### 5. Екран „Разписание"

Екранът `Schedules` е предназначен за преглед на линии и техните посоки.

- Разделя линиите по тип: автобус, тролей, трамвай, метро и нощни линии.
- Позволява избор на конкретна линия.
- Зарежда маршрутната геометрия и всички спирки по посоките на линията.
- Позволява търсене на спирка в рамките на избраната линия или глобално в списъка със спирки.
- Показва детайлни часове на тръгване по тип ден.
- Поддържа weekday/holiday разделяне на разписанията.
- Позволява отваряне на линия върху картата и фокус върху конкретна спирка.

Тук има комбиниране на два източника на данни:

- trip API от sofiatraffic.bg, когато е наличен
- bundled статично седмично разписание като fallback

### 6. Trip planner

Trip planner-ът е един от най-силните модули на приложението.

- Търси маршрут между две точки в София.
- Може автоматично да ползва текущата локация като начална точка.
- Поддържа autocomplete за адреси и локации.
- Позволява избор на дата, час и режим „тръгване в" или „пристигане до".
- Поддържа стратегии за планиране: по-малко чакане, по-малко ходене, по-малко прекачвания.
- Визуализира намерените itineraries и отделните legs в маршрута.
- Може да покаже избран маршрут директно върху картата.
- Позволява запазване на намерен маршрут за по-късно ползване.

Ограничения и особености:

- Native приложението прави директни заявки към CGM trip planner endpoint с CSRF/cookie сесия.
- Web вариантът използва локален proxy сървър, защото същите заявки не са надеждни от browser runtime.
- Сесията за trip planner се кешира ограничено във времето и се обновява при нужда.

### 7. Търсене

Търсенето в приложението е централен overlay, който комбинира няколко източника.

- Търси спирки по име и код.
- Търси линии по номер и тип.
- Търси адреси и места чрез geocoding.
- Работи с debounce, за да намалява броя на заявките.
- Резултатите могат директно да фокусират картата, да отворят спирка, да покажат линия или да стартират маршрут.

Използвани източници:

- локални транспортни данни за спирки и линии
- OpenStreetMap/Nominatim за места и адреси

### 8. Любими места

Модулът `Favorites` е добре развит и не се изчерпва с просто записване на координати.

- Позволява създаване на любими места с име и координати.
- Може да асоциира любимо място с конкретна спирка.
- Може да пази предпочитани линии за тази спирка.
- Позволява редакция, изтриване и пренареждане на любими.
- Може да маркира място като основен commute сценарий.
- Favorites панелът се отваря директно от долната action лента.

Данните се пазят локално чрез `AsyncStorage`, тоест:

- не се синхронизират между устройства
- се изтриват при деинсталация

### 9. Запазени trip planner маршрути

Освен любими места приложението пази и готови маршрути.

- Потребителят може да запази itinerary от trip planner-а.
- Всеки запазен маршрут съдържа from/to, стратегия за планиране, дата, час, summary и route GeoJSON.
- Запазен маршрут може да се отвори отново и да се покаже на картата.
- Промените по запазените маршрути се синхронизират през локален subscription механизъм.

### 10. Напомняния и известия

Има два отделни типа локални известия.

#### Напомняния за пристигане на линия

- Потребителят може да настрои напомняне за конкретна линия на конкретна спирка.
- Приложението планира локално известие няколко минути преди очакваното пристигане.
- Поддържа редакция, отказ и история на напомнянията.
- Има логика за повторно напомняне и синхронизация на reminder history.

#### Напомняния за запазен commute маршрут

- Потребителят може да зададе повтарящо се известие по дни от седмицата.
- Известието съдържа summary за маршрута и действие за отваряне на маршрута в приложението.
- При изтриване на favorite/маршрут съответните notification identifiers могат да бъдат анулирани.

Ограничения:

- Известията са локални за устройството.
- Не изискват backend, но изискват позволени notifications permissions.
- Background task логиката не е еднакво достъпна на web.

### 11. Паркинг режим

Паркинг режимът има собствен набор от действия и overlays.

#### Синя и зелена зона

- Показва паркинг зони върху картата като GeoJSON polygons.
- Определя в коя зона попада текущата локация на потребителя.
- Може да определи и зоната на dropped pin.
- Има отделен modal/settings toggle за включване и изключване на зоните.

#### Платени паркинги

- Показва каталог от паркинги и техните характеристики.
- Зарежда live наличност на свободни места от публичен parking endpoint.
- Позволява преглед на списък с паркинги и показване на конкретен паркинг върху картата.

#### Моите коли

- Потребителят може да записва автомобили с номер и име.
- Може да задава default автомобил.
- Данните се използват от parking SMS flow.

#### Parking SMS

- На Android приложението отваря SMS приложението с предварително попълнен номер и текст за избрана паркинг зона.
- Използва `Linking.openURL` с `sms:` URL схема.

Важно:

- Parking SMS функционалността работи и на iOS и Android.
- Не изисква специални permissions — отваря стандартното SMS приложение.

### 12. Настройки, поверителност и подкрепа

Модулът `settings` в момента е ограничен, но реално присъства в UI.

- Има modal за настройки.
- Включва toggle за визуализиране на паркинг зоните.
- Включва вградена политика за поверителност.
- Има отделен modal за подкрепа на проекта.

Ограничение:

- Няма обща система за persistent app settings извън конкретните локални настройки и storage за favorites/notifications.

### 13. Web вариант

Приложението има и web реализация.

- Използва `MapScreen.web.tsx` с `react-leaflet`.
- Поддържа карта, търсене, favorites и основните транспортни действия.
- За trip planner използва custom web server proxy към sofiatraffic.bg.

Ограничения на web:

- Няма native SMS интеграция.
- Няма пълния набор background/mobile възможности.
- Част от поведението е различно заради browser ограниченията.

## Функции с частична или изключена реализация

Това е важно, за да не се приема README като обещание за нещо, което кодът още не изпълнява напълно.

### Reporting / crowd reports

Има UI и service слой за докладване на:

- климатик
- запълненост
- наличие на контрольори

Но backend-ът е изключен, защото `Supabase` в момента е деактивиран. На практика това означава:

- няма реално записване на report-и
- няма realtime sync
- няма работеща crowdsourcing функционалност в production вид

### Premium / RevenueCat

Има начален service за RevenueCat, но ключовете са placeholder. Това означава:

- premium purchase flow не е готов за реална употреба
- entitlement проверките няма да работят коректно без конфигурация

### Cloud backend

В момента няма активен облачен backend за:

- автентикация
- синхронизация на данни между устройства
- reports
- Supabase realtime канали

## Локално съхранение

Приложението пази локално на устройството:

- любими места
- запазени trip planner маршрути
- локални notification настройки и идентификатори
- автомобили за parking SMS
- част от кеша за паркинг каталога

Текущият модел е local-first, без облачен sync.

## Структура на проекта

Основните директории са:

- `src/screens` - главните екрани: карта, nearby, schedules, trip planner
- `src/features/map` - карта, overlays, floating controls, mode switching
- `src/features/vehicles` - live превозни средства и анимации
- `src/features/stops` - спирки, ETA и selection state
- `src/features/search` - search overlay и резултати
- `src/features/favorites` - любими места и UI за тях
- `src/features/tripPlanner` - визуализация и helper-и за маршрути
- `src/features/parkingZones` - зони, паркинги, коли и SMS flow
- `src/features/notifications` и `src/services/notifications` - локални известия
- `src/services` - интеграции с транспортни, паркинг и storage услуги
- `scripts` - генератори на статични данни и помощни tooling скриптове

## Инсталация

Ако работиш директно в папката `SofiaGoApp`:

```bash
npm install
```

Ако работиш от repo root, има wrapper скриптове в root `package.json`, които препращат към `SofiaGoApp`.

## Стартиране

### Mobile development

```bash
npm run start
```

Полезни варианти:

```bash
npm run start -- --tunnel
npm run start -- --lan
npm run start -- --port 8082
```

### Android

```bash
npm run android
```

### iOS

```bash
npm run ios
```

### Web

```bash
npm run web
```

### Web static serving с proxy за trip planner

```bash
npm run web:serve
```

Портът по подразбиране е `3000`.

На Windows PowerShell:

```powershell
$env:PORT='8080'; npm run web:serve
```

## Data generation скриптове

Проектът включва helper скриптове за подготвяне на статични ресурси.

- `npm run generate:schedule` - генерира седмичните разписания
- `npm run generate:route-geometry` - генерира route geometry данните
- `npm run generate:parking-lots` - генерира каталога с паркинги
- `npm run generate:parking-zones` - генерира GeoJSON зоните за паркиране

## Environment бележки

В кода има следи за Supabase и RevenueCat, но в текущото състояние на проекта:

- Supabase е изключен
- RevenueCat е с placeholder ключове

Ако ще активираш тези модули, ще трябва да добавиш реални credentials и да доразвиеш backend конфигурацията.

## Често срещани проблеми

### Портът на Expo е зает

```bash
npm run start -- --port 8082
```

### Web trip planner не работи директно от browser runtime

Пусни custom сървъра с `npm run web:serve`, защото trip planner-ът на web разчита на proxy слой за CSRF/cookie сесията към sofiatraffic.bg.

### Parking SMS не работи

Провери следното:

- устройството е Android
- дадено е разрешение за SMS
- избрана е кола
- избрана е валидна зона

## Обобщение

SofiaNow вече покрива реални ежедневни сценарии за придвижване в София:

- следене на транспорт в реално време
- разглеждане на спирки и разписания
- планиране и запазване на маршрути
- локални напомняния
- търсене на адреси и спирки
- функции за паркиране и parking SMS на Android

Най-незавършените части в момента са crowdsourced reporting, облачният backend и monetization слоят.
